## 1. Template Overview

A shared component is any UI element used by more than one page. All shared components:

- Live under `src/components/shared/`
- Are prefixed with `App` (e.g. `AppButton`, `AppCard`, `AppAvatar`)
- Export a single named arrow function component
- Use Tailwind CSS v4 classes for all styling (via `cn()`)
- Define props inline or in a separate `interface` within the same file

Page-scoped sub-components follow the same structure but live inside the feature folder under `components/` and do not require the `App` prefix.

---

## 2. File Structure

### Shared component (simples)

```
src/components/shared/
└── App{ComponentName}.tsx     # Component + props interface in one file
```

### Shared component (complexo, com muitas variantes)

```
src/components/shared/
├── App{ComponentName}.tsx     # Component implementation
└── app-{component-name}.types.ts   # Interfaces e tipos exportados
```

Use um arquivo de tipos separado apenas quando a interface de props exceder ~20 linhas ou usar discriminated unions extensas.

## 3. Props Definition

### Props simples — inline no componente

```tsx
interface AppLabelProps {
  text: string
  color?: string
}
```

### Props complexas — discriminated unions para variantes

Use **discriminated unions** para componentes com variantes mutuamente exclusivas. Isso torna combinações inválidas de props um erro em tempo de compilação.

```tsx
// Inline ou em app-badge.types.ts

type AppBadgeSolid = {
  variant: "solid"
  label: string
}

type AppBadgeOutline = {
  variant: "outline"
  label: string
}

type AppBadgeIcon = {
  variant: "icon"
  icon: React.ReactNode
}

export type AppBadgeProps = AppBadgeSolid | AppBadgeOutline | AppBadgeIcon
```

### Regras

- Todos os tipos de prop são explícitos — sem `any`, sem `object`
- Props opcionais usam `?` e têm defaults via destructuring
- Callbacks são tipados com sua assinatura completa: `onClick: () => void`
- Children, quando aceitos, usam `React.ReactNode`: `children: React.ReactNode`
- Evite combinações de props booleanas que representam o mesmo conceito — use uma union `variant`
- Props de estado de carregamento/desabilitado seguem o padrão `is`: `isLoading`, `isDisabled`

## 4. Component Structure

```tsx
// src/components/shared/App{ComponentName}.tsx

import { cn } from "@/lib/utils"

interface App{ComponentName}Props {
  propA: string
  propB?: boolean
  onClick?: () => void
  className?: string
}

export function App{ComponentName}({
  propA,
  propB = false,
  onClick,
  className,
}: App{ComponentName}Props) {
  return (
    <div
      className={cn(
        "base-classes-here",
        propB && "conditional-class",
        className,
      )}
      onClick={onClick}
    >
      {propA}
    </div>
  )
}
```

### Regras

- Arrow function ou `function` declaration — ambos são aceitos; prefira `function` para componentes exportados
- **Named export** — sem `export default`
- Sem `React.FC<>` — tipar as props diretamente
- Sem lógica de negócio dentro do componente — sem `useEffect` com bodies não-triviais, sem chamadas à API
- Todo estilo via classes Tailwind e `cn()` — sem `style={{}}` inline (exceto para valores dinâmicos impossíveis de expressar com Tailwind, como larguras calculadas em runtime)
- Aceitar `className` como prop opcional para permitir override pontual pelo consumidor
- Usar tokens do design system: `bg-brand-*`, `text-success`, `bg-danger-bg`, etc.

## 5. Styling

### Usando `cn()` para classes condicionais

```tsx
import { cn } from "@/lib/utils"

// Estático
<div className="flex items-center gap-2 rounded-xl bg-card px-4 py-2" />

// Condicional
<div
  className={cn(
    "flex items-center gap-2 rounded-xl px-4 py-2",
    variant === "solid" && "bg-brand-500 text-white",
    variant === "outline" && "border border-brand-500 text-brand-500",
    isDisabled && "opacity-50 cursor-not-allowed",
    className,
  )}
/>
```

### Variantes com mapa de classes

Para componentes com múltiplas variantes, use um objeto de configuração:

```tsx
const VARIANT_CLASSES = {
  default:  "bg-muted text-muted-foreground",
  success:  "bg-success-bg text-success",
  warning:  "bg-warning-bg text-warning",
  danger:   "bg-danger-bg text-danger",
  info:     "bg-info-bg text-info",
} satisfies Record<string, string>

// Uso
<span className={cn("rounded-full px-2.5 py-0.5 text-xs font-medium", VARIANT_CLASSES[variant])} />
```

### Regras de estilo

- Usar tokens Tailwind do design system em vez de cores hardcoded (`bg-brand-500`, não `bg-[#7551FF]`)
- Cores hardcoded via `bg-[#hex]` são permitidas apenas quando não existe token equivalente
- Variantes de dark mode via seletor `.dark` já configurado em `index.css`; usar prefixo `dark:` no Tailwind quando necessário
- Animações via classes Tailwind (`animate-spin`, `animate-pulse`, `animate-ping`)

## 6. Complete Example

### `AppPill` — um badge de label simples

```tsx
// src/components/shared/AppPill.tsx

import { cn } from "@/lib/utils"

type Variant = "default" | "success" | "warning" | "danger"

interface AppPillProps {
  label: string
  variant?: Variant
  onClick?: () => void
  className?: string
}

const VARIANT_CLASSES: Record<Variant, string> = {
  default: "bg-muted text-muted-foreground",
  success: "bg-success-bg text-success",
  warning: "bg-warning-bg text-warning",
  danger:  "bg-danger-bg text-danger",
}

export function AppPill({
  label,
  variant = "default",
  onClick,
  className,
}: AppPillProps) {
  return (
    <span
      className={cn(
        "inline-flex items-center rounded-full px-2.5 py-0.5 text-xs font-medium",
        VARIANT_CLASSES[variant],
        onClick && "cursor-pointer hover:opacity-80 transition-opacity",
        className,
      )}
      onClick={onClick}
    >
      {label}
    </span>
  )
}
```

**Uso:**

```tsx
<AppPill label="Disponível" variant="success" />
<AppPill label="Em revisão" variant="warning" />
<AppPill label="Vendido" variant="danger" />
```

## 7. Checklist

Antes de submeter um novo componente, verifique:

- [ ] Arquivo em `src/components/shared/App{ComponentName}.tsx`
- [ ] Named export — sem `export default`
- [ ] Sem `React.FC<>` — props tipadas diretamente
- [ ] Props explicitamente tipadas (inline ou em arquivo separado)
- [ ] Discriminated union usada para props de variante em vez de múltiplos booleanos
- [ ] Todo estilo via `cn()` + classes Tailwind — sem `style={{}}` inline (exceto casos excepcionais)
- [ ] Tokens do design system usados (não cores hardcoded) quando disponíveis
- [ ] `className` aceito como prop opcional para override externo
- [ ] Sem lógica de negócio, sem chamadas à API, sem dispatch ao Zustand
- [ ] Sem `console.log`
- [ ] Sem `any` types
- [ ] Testado em light mode e dark mode
